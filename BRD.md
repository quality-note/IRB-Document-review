# BRD (Business Requirements Document)
## 임상연구 심의문서 검토 지원 시스템

- **작성일**: 2026-07-06 (최초 작성), 이후 반영된 모든 수정사항 포함하여 최종 정리
- **버전**: v1.0 (MVP, 실사용 반영 완료 단계)
- **문서 상태**: 현재 구현 상태를 기준으로 작성된 최종본

---

## 1. 배경 및 필요성

IRB(기관생명윤리위원회) 행정간사 및 연구지원팀은 연구자가 제출한 심의서류(연구계획서, 임상적 성능시험계획서, 설명문 및 동의서, 결과보고서 등)를 심의위원회 상정 전에 사전 검토한다. 이 사전 검토는 현재 수작업으로 이루어지며 다음과 같은 문제가 있다.

- 문서별 필수 기재사항 누락 여부를 사람이 매번 수기로 대조해야 함
- 여러 문서 간(연구계획서/동의서/결과보고서 등) 핵심 정보(연구 제목, 기간, 대상자 수, 검체 보관기간 등)의 불일치를 놓치기 쉬움
- 관련 법령·가이드라인·기관 SOP·과거 심의사례를 검토 시점마다 다시 찾아봐야 함
- 연구자에게 보낼 보완요청 공문(행정의견)을 매번 새로 작성해야 함

이를 보조하기 위해 문서 업로드 → 사전 점검 → 결과 확인 → 행정의견 작성까지 이어지는 내부 업무 지원 시스템을 구축한다.

### 1.1 핵심 원칙 (반드시 지켜야 하는 제약)

본 시스템은 **연구의 승인 여부, 규제 적합 여부, IRB 심의 대상 여부, 식약처 승인 대상 여부를 최종 판단하지 않는다.** 이는 법적·조직적 책임 소재상 반드시 지켜야 하는 제약이며, 이번 구현 전 기간에 걸쳐 다음과 같이 반영되었다.

- 화면·결과 문구에 "적합/부적합", "승인 가능/불가", "심의 대상 아님" 등 단정적 표현을 사용하지 않는다.
- "확인 필요", "보완 권고", "검토 필요", "근거 확인 필요", "참고 의견", "담당자 최종 확인 필요" 등 보조적 표현만 사용한다.
- 위험도(높음/중간/낮음/참고)는 **행정검토 우선순위**를 의미하며, 연구의 법적 적합성·승인 가능성과 무관함을 화면에 항상 안내한다.
- 모든 AI 분석 결과에는 "담당자 확인 필요" 표시가 붙는다.
- 근거자료는 "근거 후보"로만 제시되며, 담당자가 최종 판단·선택한다.
- 연구자에게 자동으로 문서/의견을 발송하는 기능은 만들지 않는다. 모든 행정의견 초안은 담당자가 수정 후 확정하는 구조다.

## 2. 비즈니스 목표

| 목표 | 설명 |
|---|---|
| 사전 검토 표준화 | 필수 기재사항 누락, 문서 간 불일치를 일관된 기준으로 자동 정리 |
| 검토 이력 관리 | 접수번호 단위로 검토 상태·이력·담당자 메모를 한 곳에서 추적 |
| 근거자료 접근성 향상 | 법령·가이드라인·SOP·과거 심의사례를 검색·연결해 검토 근거를 빠르게 확인 |
| 행정 소통 효율화 | 연구자 회신용 공문(행정의견) 초안을 자동 생성하고 담당자가 다듬어 사용 |
| 향후 확장성 확보 | 실제 LLM 분석, Supabase 백엔드, 전자결재/발송 연동을 위한 구조 선확보 |

## 3. 이해관계자

| 역할 | 관련 업무 |
|---|---|
| 행정간사 (주 사용자, 예: 백주환) | 검토 건 등록, 문서 업로드, AI 사전 검토 실행, 행정의견 초안 작성/회신 준비 |
| 행정간사 (보조, 예: 오은하) | 동일 권한으로 업무 분담 |
| 심의위원회 | 시스템 산출물을 참고자료로 활용하되 최종 심의·의결 수행 (시스템 사용자는 아님) |
| 기관/책임자 | 규정 준수 여부에 대한 최종 판단 주체 |

현재 시스템 로그인 계정은 위 두 행정간사 계정으로 한정되어 있다.

## 4. 범위

### 4.1 포함 범위 (In Scope, 구현 완료)

- 검토 과제(접수번호 단위) 등록·조회·상태 관리
- 심의서류 업로드 및 문서 종류 분류
- 업로드된 문서 종류를 기준으로 한 mock AI 사전 검토(필수항목 누락/문서 간 불일치/개인정보·검체 관리/연구자 확인 질문/규정 검토 참고)
- 재검토(지식베이스 변경 등을 반영해 검토 결과를 다시 생성)
- 근거자료(지식베이스) 등록·검색·수정·삭제 및 실제 첨부파일 연결
- 문체 선택형 행정의견 초안 자동 생성 및 편집, 저장 이력 관리
- 대시보드 현황 요약
- 로그인/로그아웃/비밀번호 변경 (내부용 1차 접근 통제)
- 인쇄(브라우저 인쇄) 및 Word 호환 파일(mock) 내보내기

### 4.2 제외 범위 (Out of Scope, 이번 단계 미포함)

- 실제 LLM API를 통한 문서 내용 분석 (현재는 업로드 문서 종류 기반의 규칙형 mock 분석)
- 실제 백엔드/데이터베이스(Supabase) 연동 — 현재는 브라우저 메모리 상태이며 새로고침 시 이번 세션에서 등록한 신규 데이터는 초기화됨 (단, 소스에 기본값으로 박아넣은 시드 데이터는 유지)
- 서버 기반의 진짜 인증/권한 관리 (현재는 브라우저 저장소 기반의 1차 접근 통제 수준)
- 연구자에게 문서/의견을 직접 발송하는 기능
- 실제 PDF/Word 생성 엔진을 통한 정식 문서 출력 (현재는 브라우저 인쇄 및 HTML 기반 mock Word 파일)
- OCR/문서 본문 자동 추출 (`extractedText` 필드는 구조만 선반영, 실제 추출 로직 없음)

## 5. 성공 기준

- 신규 검토 건 등록부터 문서 업로드, AI 사전 검토, 결과 확인, 행정의견 작성까지 담당자가 이탈 없이 한 흐름으로 완료할 수 있다.
- 실제 업로드하지 않은 문서 종류에 대한 검토 항목이 노출되지 않는다 (실사용 피드백으로 확인·수정 완료).
- 지식베이스에서 근거자료를 삭제/수정하면, 재검토 시 관련 검토 항목·근거 표시가 실제 상태에 맞게 갱신된다.
- 모든 화면 및 출력물에 판단 주체가 담당자·위원회·기관이라는 점과 위험도의 의미(행정 우선순위)가 명시되어 있다.
- 담당 직원 2인 계정으로만 로그인 가능하다.

## 6. 리스크 및 가정

| 구분 | 내용 |
|---|---|
| 보안 리스크 | 현재 로그인은 프론트엔드 코드/브라우저 저장소 기반이라, 개발자 도구로 소스를 열람하면 계정 정보가 노출될 수 있음. 진짜 보안이 필요해지면 Supabase Auth 등 서버 인증 도입이 선행되어야 함 |
| 데이터 영속성 리스크 | 브라우저를 새로고침해도 유지되어야 하는 데이터는 반드시 소스 코드의 시드 데이터로 박아 넣어야 하며, 그 외 세션 중 입력값은 백엔드 연동 전까지 새로고침 시 초기화됨 |
| 분석 신뢰도 가정 | 현재 AI 분석은 규칙 기반 mock이며 실제 문서 내용을 읽지 않음. 실제 LLM 연동 전까지는 "참고용 흐름 검증" 목적으로만 사용해야 함 |
| 조직 변경 가정 | 검토 담당자는 현재 2인(백주환, 오은하)으로 가정. 인원 변경 시 계정·목록을 코드에서 갱신해야 함 |

## 7. 향후 로드맵 (제안)

1. **Supabase 연동**: `ReviewDataContext.tsx`의 각 CRUD 함수 내부를 Supabase 클라이언트 호출로 교체 (테이블 스키마는 `src/types` 인터페이스와 1:1 대응 설계됨)
2. **실제 인증(Supabase Auth 등) 연동**: `authService.ts` 내부만 교체
3. **실제 LLM API 연동**: `aiReviewService.ts`의 `analyzeDocuments()`/`generateOpinionDraft()` 내부만 교체 (입출력 타입 고정)
4. **실제 파일 저장소 연동**: 현재 `public/knowledge-base`, `public/project-documents` 정적 파일 방식 → Supabase Storage 등으로 교체
5. **정식 Word/PDF 생성**: 현재 mock 내보내기를 `docx`/`pdf-lib` 등 라이브러리 기반으로 교체
